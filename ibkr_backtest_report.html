pieces — each user's data lives in their own GitHub fork; only the UI is shared. ``` ┌──────────────────────────┐ │ kalshitradingbot │ Netlify Next.js — read-only window │ .netlify.app │ into the user's bot. NO bot runs here. │ (kalshi-dashboard repo) │ └────────────┬─────────────┘ │ user creds encrypted with AES-256-GCM │ (ENCRYPTION_KEY in Netlify env, never in code) ▼ ┌──────────────────────────┐ │ Clerk privateMetadata │ Per-user storage. Only the server │ - kalshiApiKey │ can read it (private metadata), │ - kalshiPrivateKey │ and every field is AES-encrypted. │ - githubPat │ │ - githubRepo │ └────────────┬─────────────┘ │ /api/bot/start dispatches workflow │ + pushes sealed-box Actions secrets ▼ ┌──────────────────────────┐ │ User's fork of │ The actual bot lives here. The │ KalshiTradingBot │ fork should stay PUBLIC (unlimited │ .github/workflows/ │ Actions minutes). Kalshi secrets │ bot.yml │ are stored as encrypted Actions │ kalshi_btc_paper_bot.js│ secrets — private even on a public │ dashboard/ │ repo. │ btc_paper_state.json │ └──────────────────────────┘ ``` ### Lifecycle of one bot run 1. User signs in to dashboard via Clerk. 2. Dashboard `/api/user` POST encrypts Kalshi key + PEM, writes to Clerk privateMetadata. 3. User connects GitHub (fine-grained PAT scoped to ONE repo with Contents + Actions + Secrets). 4. User picks their fork of `KalshiTradingBot` from the repo list. 5. Dashboard `/api/bot/start` validates PAT scope, then: - libsodium sealed-box encrypts Kalshi creds against the repo's public key - PUTs them as `KALSHI_API_KEY` / `KALSHI_PRIVATE_KEY` Actions secrets - POSTs a workflow_dispatch event to `bot.yml` 6. GitHub Actions runner spins up, writes PEM to disk, runs `node kalshi_btc_paper_bot.js`. 7. Every 30s the bot scans markets; every ~90s it commits `dashboard/btc_paper_state.json` back to the user's fork (force-with-lease amend — single rolling commit, no history pollution). 8. Bot self-terminates at 5h50m; cron `0 */6 * * *` starts the next run cleanly. 9.
